# 薪酬与发薪流程 (Payroll & Billing Flow)

在 Extend Global (EG) 平台中，薪酬计算与账单生成是系统最核心的定时任务（Cron Job）与财务引擎协同工作的产物。本文档详细描述了每月从数据截账到最终资金下发的完整工作流。

## 1. 薪酬周期概述

EG 平台默认采用**按月（Monthly）**的薪酬周期。

* **计薪周期 (Payroll Period)**：通常为自然月（如 3月1日 - 3月31日）。
* **截账日 (Cut-off Date)**：通常设定在每月的 20 日或 25 日。在此日期之后提交的请假或报销，将顺延至下一个计薪周期处理。
* **发薪日 (Pay Date)**：员工实际收到工资的日期，通常为当月最后一天或次月 5 日（取决于当地国家法规与 Vendor 的要求）。

## 2. 核心工作流步骤

每月的发薪流程可以划分为以下五个连续的阶段：

### 2.1 阶段一：数据准备与截账 (Cut-off)
在这个阶段，系统需要收集所有影响当月最终薪酬的变量数据。

1. **基础薪资确认**：提取所有处于 `Active` 状态员工的基础月薪（Base Salary）。如果员工在当月中旬入职，系统会根据其 `startDate` 按当月工作日比例自动折算（Proration）。
2. **请假扣款计算**：汇总该员工在当月已获批（Approved）的无薪假（Unpaid Leave）天数，并计算相应的扣款金额。
3. **报销汇总**：汇总该员工在当月已获批且标记为随工资发放的费用报销单（Reimbursements）。
4. **津贴与奖金**：添加客户 HR 手动录入的一次性奖金（Bonus）或固定津贴（Allowance）。

### 2.2 阶段二：薪酬计算 (Payroll Run)
系统将收集到的原始数据交由 `calculationService` 进行处理。

1. **生成 Payroll Items**：为每位员工生成详细的薪资明细项（Payroll Items），包括 Gross Pay、Deductions、Net Pay 以及雇主社保成本（Employer Contributions）。
2. **汇率锁定**：系统通过 `exchangeRateService` 获取当月的基准汇率（Mid-market Rate），并将所有当地货币（如 EUR）计算的成本转换为系统的结算货币（通常为 USD）。
3. **状态标记**：生成的 Payroll Run 状态标记为 `Draft`，等待 Admin 审核。

### 2.3 阶段三：双层发票生成 (Invoice Generation)
当 Payroll Run 确认无误后，系统调用 `dualLayerInvoiceService`（双层发票引擎）。

1. **按客户分组**：将同一客户下的所有员工薪酬数据打包。
2. **生成 Layer 1 发票 (EG → CP)**：加上 EG 对 CP 的底层服务费，生成草稿发票。
3. **生成 Layer 2 发票 (CP → Client)**：加上 CP 对客户的加价服务费，生成草稿发票。
4. **发票发布**：Admin 审核通过后，发票状态变为 `Published`，系统自动生成白标 PDF 并通过邮件发送给相应的 CP 与 Client。

### 2.4 阶段四：客户付款与资金流转 (Fund Flow)
这是确保资金链安全的关键步骤，完全依赖于人工确认。

1. **Client 支付**：终端客户将 Layer 2 发票金额汇入 CP 的银行账户。CP 在门户中标记发票为 `Paid`。
2. **CP 支付**：CP 在门户中勾选对应的 Layer 1 发票，点击 "Pay from Wallet"。系统扣除 CP 钱包余额，将 Layer 1 发票标记为 `Paid`。
3. **前置条件检查**：只有当 Layer 1 发票被全额支付后，系统才允许进入最终的资金下发阶段。这是防止 EG 垫资风险的硬性合规要求。

### 2.5 阶段五：资金下发与供应商对账 (Payout & Reconciliation)
EG 确认收到 CP 的款项后，执行最终的资金动作。

1. **净薪资下发**：EG 财务团队通过全球支付网络（如 Wise, Airwallex）或当地 Vendor，将税后净薪资（Net Pay）打入员工的个人银行账户。
2. **税费缴纳**：将代扣的个人所得税与雇主社保缴纳至当地政府机构。
3. **供应商账单对账**：如果是通过 Vendor 落地，Vendor 会发来最终的实际账单（Vendor Bill）。系统调用 `reconciliationEngine`（双币种对账引擎），将系统生成的预估成本发票与 Vendor 的实际账单进行匹配，计算并记录任何因汇率波动或社保基数调整产生的差异（Variance）。

## 3. 异常场景处理

在实际操作中，经常会遇到以下异常场景，系统提供了相应的处理机制：

### 3.1 离职结算 (Off-cycle Payroll)
如果员工在月中突然离职，系统支持生成非周期的离职结算单（Off-cycle Payroll）。该流程会立即计算截至离职日的薪资、未休年假折现以及可能的遣散费，并单独生成一份即时账单，不等待月末的统一发薪。

### 3.2 错发与红字冲抵 (Credit Notes)
如果发票已发布甚至已支付后，发现薪资计算错误（例如少算了某项扣款），系统严禁直接修改历史发票。此时必须通过生成红字发票（Credit Note）来进行冲抵，并在下个月的账单中体现这笔调整（Adjustment）。

通过这种结构化、分阶段的处理流程，Extend Global 平台确保了全球成千上万名员工薪酬计算的准确性、发票生成的高效性以及资金流转的安全性。
