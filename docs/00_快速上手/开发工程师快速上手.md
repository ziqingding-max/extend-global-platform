# 开发工程师 (Dev) 快速上手指南

欢迎加入 Extend Global (EG) 研发团队！本项目是一个基于全栈 TypeScript 的 Monorepo。我们的技术栈核心是 **React + Vite + tRPC + Drizzle ORM + SQLite**。

本指南将帮助你在 1 小时内把项目跑起来，并理解核心的代码流转机制。

## 1. 本地环境初始化

### 1.1 前置要求
* Node.js (推荐 v20+)
* pnpm (必须使用 pnpm，不建议使用 npm/yarn 以避免依赖幽灵问题)
* Git

### 1.2 启动步骤
1. **克隆代码库**
   ```bash
   git clone git@github.com:your-org/extend-global-platform.git
   cd extend-global-platform
   ```

2. **安装依赖**
   ```bash
   pnpm install
   ```

3. **配置环境变量**
   ```bash
   cp .env.example .env
   ```
   *注意：本地开发时，`DATABASE_URL` 默认指向根目录下的 `local.db`，无需额外配置真实的第三方 API Key 即可启动基础服务。*

4. **初始化数据库 (Drizzle)**
   ```bash
   # 根据 schema.ts 生成本地 SQLite 数据库文件并推送表结构
   pnpm run db:push
   
   # (可选) 注入测试种子数据，方便本地调试
   pnpm run db:seed
   ```

5. **启动开发服务器**
   ```bash
   # 并发启动前端 Vite 和后端 tRPC Server
   pnpm run dev
   ```

启动成功后，你可以访问：
* 前端（Admin Portal 默认入口）: `http://localhost:5173`
* 后端 API: `http://localhost:5000/trpc`

## 2. 必读的架构文档

在动手写第一行代码前，请**必须**阅读以下三篇技术文档，否则你很可能会破坏系统的核心逻辑：

1. **[代码结构与分层](../05_技术架构/代码结构与分层.md)**：了解前端三个 Portal 的物理隔离，以及后端 Router 与 Service 的职责边界。
2. **[数据库Schema详解](../05_技术架构/数据库Schema详解.md)**：这是整个系统的地基。特别是 `invoices`, `channelPartnerWallets` 和 `users` 这几张表的关系。
3. **[认证与权限体系](../05_技术架构/认证与权限体系.md)**：**极其重要！** 了解如何在 tRPC 路由层编写带有数据隔离（Data Isolation）的查询，防止出现越权漏洞（IDOR）。

## 3. 日常开发工作流 (Workflow)

### 3.1 如何新增一个 API 接口？
假设你需要新增一个“查询员工详情”的接口。

1. **后端：定义 Router**
   打开 `server/routers/employees.ts`，使用 tRPC 定义接口，并加上权限中间件：
   ```typescript
   import { z } from 'zod';
   import { cpAdminProcedure } from './_core/procedures';
   import { db } from '../services/db';
   import { employees } from '../../drizzle/schema';
   import { eq, and } from 'drizzle-orm';

   export const getEmployeeDetail = cpAdminProcedure
     .input(z.object({ employeeId: z.number() }))
     .query(async ({ ctx, input }) => {
       // 强制数据隔离：必须通过 ctx.user 校验归属权
       const query = db.select().from(employees).where(
         and(
           eq(employees.id, input.employeeId),
           // 假设员工表有关联的客户ID，客户表有关联的 CP ID，这里需要做 Join 或子查询校验
           // 为了演示简便，假设员工表直接有 cpId
           ctx.user.role === 'admin' ? undefined : eq(employees.channelPartnerId, ctx.user.channelPartnerId)
         )
       );
       return await query;
     });
   ```

2. **前端：调用 Hook**
   由于使用了 tRPC，你不需要写任何 axios 请求或定义 TypeScript 接口。直接在 React 组件中调用：
   ```tsx
   import { trpc } from '../../utils/trpc';

   function EmployeeProfile({ id }) {
     // 这里的 data 会被自动推导为 Employee 类型！
     const { data, isLoading } = trpc.employees.getEmployeeDetail.useQuery({ employeeId: id });
     
     if (isLoading) return <Spinner />;
     return <div>{data.baseSalary}</div>;
   }
   ```

### 3.2 如何修改数据库表结构？
1. 打开 `drizzle/schema.ts`。
2. 找到对应的表（或新增表），添加字段。
   ```typescript
   export const employees = sqliteTable('employees', {
     id: integer('id').primaryKey({ autoIncrement: true }),
     // ... 其他字段
     newField: text('new_field'), // 新增字段
   });
   ```
3. 运行生成迁移文件的命令：
   ```bash
   pnpm run db:generate
   ```
4. 运行推送命令将变更应用到本地 `local.db`：
   ```bash
   pnpm run db:push
   ```
5. 提交代码时，**必须**将 `drizzle/` 目录下新生成的 `.sql` 迁移文件一并提交。

## 4. 开发红线 (Red Lines)

1. **严禁直接操作金额浮点数**：在进行财务计算时，严禁使用原生的 `a + b`（会产生 `0.1 + 0.2 = 0.30000000000000004` 的问题）。必须使用项目内置的精度计算库（如 `decimal.js` 或自定义的 `currencyUtils`）。
2. **严禁无事务扣款**：任何涉及钱包余额（`channelPartnerWallets`）扣减或发票状态变更的操作，**必须**包裹在 Drizzle 的事务（Transaction）中，并对钱包记录使用 `FOR UPDATE` 悲观锁。
3. **严禁硬编码品牌信息**：在开发 App Portal 或 Worker Portal 的前端页面时，绝对不能硬编码 "Extend Global" 字样或 Logo。必须通过 `useCpBranding()` Hook 获取当前环境的白标配置。

## 5. 调试技巧

* **查看生成的 SQL**：在开发环境中，Drizzle ORM 会将执行的 SQL 语句打印在控制台。遇到复杂查询时，先看控制台打印的 SQL 是否符合预期。
* **白标多域名调试**：在本地测试 CP 白标功能时，你可以修改本机的 `/etc/hosts` 文件，将 `acme.localhost` 和 `hr.localhost` 指向 `127.0.0.1`，然后在浏览器中通过 `http://acme.localhost:5173` 访问，以触发白标 Hook 的解析逻辑。
