# CP 白标机制 (White-label Mechanism)

Extend Global (EG) 平台的核心商业价值之一是赋能渠道合作伙伴 (CP)，使其能够以自有品牌向终端客户提供全球名义雇主 (EOR) 服务。本文档详细解析系统如何通过通配符子域名、动态主题注入以及文档模板渲染，实现全链路的“白标定制”。

## 1. 白标机制的设计目标

在 B2B2B 模式下，终端客户 (Client) 和员工 (Worker) 不应感知到 EG 的存在。因此，白标机制必须实现以下三个层面的完全隔离：

1. **视觉隔离**：门户界面的域名、Logo、主色调、Favicon 必须替换为 CP 的品牌资产。
2. **文档隔离**：系统生成的劳动合同、发票 (PDF)、工资单等正式文件，必须使用 CP 的抬头和样式。
3. **通信隔离**：系统发送的所有邮件通知，其发件人名称、邮件模板样式必须与 CP 品牌一致。

## 2. 域名与前端路由架构

白标机制的基础是动态域名解析与前端路由的无缝配合。

### 2.1 通配符子域名配置
在 DNS 层，系统配置了 `*.extendglobal.ai` 的通配符 A 记录，指向部署服务器的 IP 地址。
在 Nginx 层，配置了一个专门的 `server` 块监听 `*.extendglobal.ai`，拦截所有非 `admin.` 和 `app.` 的子域名请求，并将其统一代理至 Node.js 服务。

### 2.2 前端品牌上下文注入
当用户在浏览器中输入 `acme.extendglobal.ai` 时：
1. React SPA 启动，`useCpBranding` Hook 拦截路由初始化。
2. 提取 URL 中的子域名 `acme`，向后端 API `/api/branding/acme` 发起请求。
3. 后端查询 `channelPartners` 表，如果匹配成功，返回该 CP 的品牌配置（Logo URL、主色调 HEX 值、公司名称）。
4. 前端将这些配置写入全局状态（如 React Context 或 Redux）。
5. 页面组件读取全局状态：
   * 将导航栏的 Logo 替换为 CP 的 Logo。
   * 动态生成一段包含 CSS 变量的 `<style>` 标签插入 `<head>`，覆盖 TailwindCSS 或 Ant Design 的默认主色调。
   * 动态修改 `document.title` 和 `<link rel="icon">`。

如果子域名不存在或无效，前端会重定向到一个通用的 404 错误页面，避免暴露系统底层结构。

## 3. 文档与 PDF 渲染引擎

系统中的发票、合同和工资单均由后端的 PDF 渲染引擎动态生成。

### 3.1 模板分层设计
PDF 生成采用了 `HTML-to-PDF` 的技术方案（通过 Puppeteer 或类似工具）。模板设计分为两层：
* **结构层 (Base Template)**：定义了文档的骨架，如表格布局、数据排版，这部分对所有 CP 都是通用的。
* **样式层 (Brand Layer)**：定义了页眉、页脚、Logo 位置和主题颜色。

### 3.2 动态渲染流程
当系统需要生成一份 Layer 2 发票给终端客户时：
1. `invoicePdfService` 提取发票的业务数据。
2. 查询该发票所属的 CP 品牌配置。
3. 将业务数据和品牌配置同时注入到 Handlebars (或类似模板引擎) 中。
4. 模板引擎将 CP 的 Logo 插入页眉，将表格标题背景色设置为 CP 的主色调，并在页脚插入 CP 的公司注册地址。
5. 最终渲染出的 PDF 文件完全呈现为 CP 的官方文档，随后上传至阿里云 OSS 存储。

## 4. 邮件通知系统白标化

系统触发的自动化邮件（如入职邀请、发票催款、密码重置）同样遵循白标原则。

### 4.1 邮件模板引擎
后端的 `cpEmailService` 负责处理白标邮件的发送。所有的邮件模板（HTML 格式）都预留了品牌变量占位符。

### 4.2 发件人与内容替换
发送邮件时，服务会执行以下替换：
* **发件人名称 (Sender Name)**：从 "Extend Global" 替换为 "[CP Company Name] HR" 或 "[CP Company Name] Finance"。
* **发件地址 (From Address)**：由于反垃圾邮件策略的限制，系统通常无法直接使用 CP 的自有域名发送邮件。标准的做法是使用 `no-reply@extendglobal.ai`，但将 "Reply-To" 地址设置为 CP 提供的客服邮箱。
* **邮件正文**：邮件头部插入 CP Logo，所有操作按钮的颜色调整为 CP 主色调，所有链接均指向 CP 的专属子域名（如 `https://acme.extendglobal.ai/portal/...`）。

## 5. 运营配置与审核流程

为了防止滥用并保证视觉质量，CP 的白标配置需经过 EG 运营团队的审核。

1. **提交申请**：CP 管理员在门户中上传高清透明背景的 Logo，选择品牌主色调，并提交期望的子域名前缀。
2. **后台审核**：EG 运营人员在 Admin Portal 中预览渲染效果，确认 Logo 清晰、颜色搭配合理且子域名未被占用后，点击“批准”。
3. **即时生效**：配置一旦批准，系统立即更新数据库。由于前端品牌数据存在短暂缓存，通常在几分钟内，终端客户即可看到全新的品牌界面。

通过上述机制，Extend Global 实现了技术底座的复用与商业品牌的隔离，为渠道合作伙伴提供了开箱即用的 SaaS 级白标体验。
