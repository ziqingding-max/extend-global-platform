# 常见问题与排查指南 (Troubleshooting Guide)

本文档汇总了 Extend Global (EG) 平台在生产环境中最常见的故障场景及其排查步骤。当收到告警或用户反馈时，运维人员应首先参照本文档进行初步诊断。

## 1. 应用层故障

### 1.1 白标页面加载白屏或显示 EG 默认品牌

**现象**：CP 用户通过其专属子域名（如 `acme.extendglobal.ai`）访问时，页面白屏或者显示了 Extend Global 的默认 Logo。

**排查步骤**：
1. 检查 DNS 解析是否正确。在服务器上执行 `nslookup acme.extendglobal.ai`，确认其解析到了正确的服务器 IP。
2. 检查 Nginx 配置。确认 `server_name` 中包含了 `*.extendglobal.ai` 的泛域名匹配。
3. 检查后端日志。搜索 `useCpBranding` 或 `getBrandingBySubdomain` 相关的错误。最常见的原因是该 CP 在 `channelPartners` 表中的 `subdomain` 字段值与实际访问的子域名不匹配（注意大小写）。
4. 检查 CORS 配置。如果后端 API 和前端不在同一个域名下，确认后端的 CORS 白名单中包含了该子域名。

### 1.2 发票 PDF 生成失败

**现象**：Admin 点击"生成 PDF"后，系统报错或返回空白 PDF。

**排查步骤**：
1. 检查容器内存。PDF 生成（通常使用 Puppeteer 或类似库）是内存密集型操作。如果服务器 RAM 不足，可能导致 Chromium 子进程被 OOM Killer 杀死。通过 `docker stats` 查看容器的内存使用率。
2. 检查字体文件。如果 PDF 中包含中文或日文等 CJK 字符，容器内必须安装对应的字体包（如 `fonts-noto-cjk`）。
3. 检查白标模板。PDF 模板中引用的 Logo URL 是否可访问。如果 Logo 存储在 S3 上，确认 S3 Bucket 的权限策略允许公开读取。

### 1.3 邮件发送失败

**现象**：员工入职邀请邮件或发票通知邮件未送达。

**排查步骤**：
1. 检查 `.env` 中的 SMTP 配置（`SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`）是否正确。
2. 检查邮件服务商（如 SendGrid）的控制台，查看是否有被退信（Bounce）或被标记为垃圾邮件（Spam）的记录。
3. 确认发件人域名已通过 SPF 和 DKIM 验证。如果使用白标发件人（如 `noreply@acme-hr.com`），该域名的 DNS 必须配置了正确的 SPF 和 DKIM 记录。

## 2. 数据库层故障

### 2.1 SQLite 数据库锁定 (Database is locked)

**现象**：后端日志中频繁出现 `SQLITE_BUSY: database is locked` 错误。

**原因分析**：SQLite 是一个文件级数据库，在高并发写入时，会出现锁竞争。这在多个请求同时尝试写入钱包流水或更新发票状态时尤为常见。

**排查与解决**：
1. 确认 SQLite 的 WAL (Write-Ahead Logging) 模式已开启。在应用启动时执行 `PRAGMA journal_mode=WAL;`。WAL 模式允许读写并发，能显著降低锁冲突。
2. 确认所有写操作都包裹在事务中，且事务的持续时间尽可能短。避免在事务中执行耗时的外部 API 调用。
3. 如果并发量持续增长，考虑将数据库迁移至 PostgreSQL 或 MySQL。

### 2.2 数据库迁移失败

**现象**：执行 `pnpm run db:push` 或 `pnpm run db:migrate` 时报错。

**排查步骤**：
1. 检查错误信息。最常见的原因是新增了一个 `NOT NULL` 字段但没有提供默认值，导致已有数据行无法满足约束。
2. 如果是生产环境，**绝对不能**直接修改 `production.db`。应先在本地复现问题，修复迁移脚本后重新部署。
3. 在执行迁移前，务必先备份数据库文件。

## 3. 资金与财务故障

### 3.1 钱包余额与流水不一致

**现象**：`channelPartnerWallets` 表中的 `balance` 字段值，与 `walletTransactions` 表中所有该钱包流水的 `SUM(amount)` 不一致。

**严重等级**：**P0 - 致命**。这意味着系统的资金数据可能已经出错。

**排查步骤**：
1. 立即暂停该 CP 的所有钱包操作（如果系统支持，将 CP 状态设为 `suspended`）。
2. 编写 SQL 查询，对比两个表的数据：
   ```sql
   SELECT 
     w.id, w.balance,
     (SELECT COALESCE(SUM(amount), 0) FROM walletTransactions WHERE walletId = w.id) as calculated_balance
   FROM channelPartnerWallets w
   WHERE w.balance != (SELECT COALESCE(SUM(amount), 0) FROM walletTransactions WHERE walletId = w.id);
   ```
3. 检查 `walletTransactions` 中是否存在没有对应 `invoiceId` 的异常流水。
4. 检查审计日志 (`auditLogs`)，查看是否有 Admin 手动修改了余额。
5. 确认差异原因后，由财务负责人签字确认，再由 Admin 手动创建一笔 `adjustment` 类型的流水来修正余额。

### 3.2 发票金额为 0 或负数

**现象**：生成的发票中，某些员工的服务费金额为 $0 或出现负数。

**排查步骤**：
1. 检查该员工对应的定价规则（`cpPricingRules` 或 `customerPricing`）是否已正确配置。
2. 如果使用"按比例收费"，检查该员工当月的总用工成本是否为 0（可能是因为员工全月无薪假）。
3. 如果出现负数，检查是否有未正确处理的红字发票（Credit Note）或调整项（Adjustment）被错误地叠加。

通过系统化地遵循上述排查指南，运维团队可以在最短时间内定位并解决生产环境中的常见故障，最大限度地降低对 CP 和终端客户的业务影响。
