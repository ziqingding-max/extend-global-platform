# 测试工程师 (QA) 快速上手指南

欢迎加入 Extend Global (EG) 质量保障团队！作为一个处理全球薪资和资金流转的金融级系统，你的工作是守护公司的生命线。在这里，一个看似微小的 Bug（如汇率取错了一位小数），都可能导致公司面临巨额的财务损失或合规罚款。

本指南将帮助你快速理解系统的测试重点和测试环境的搭建。

## 1. 测试环境与账号矩阵

为了完整测试 B2B2B 的多层级业务，你不能只用一个账号测到底。你需要准备（或让开发帮你生成）一套**账号矩阵**，并在不同的浏览器（或无痕窗口）中同时登录，以模拟多方交互。

### 标准测试账号矩阵
1. **Admin 账号** (登录 `localhost:5173` 或 Admin 测试环境)
   * 角色：上帝视角，负责审批、配置底层规则。
2. **CP Admin 账号** (登录 `acme.localhost:5173`，假设 Acme 是一个 CP)
   * 角色：渠道商，负责配置对客加价、支付 Layer 1 账单。
3. **Client Admin 账号** (登录同一个 CP 域名)
   * 角色：终端客户 HR，负责发起入职、支付 Layer 2 账单。
4. **Worker 账号** (通过邮件邀请链接激活)
   * 角色：员工，负责填写入职资料、申请请假。

**💡 提示**：由于系统通过 Cookie 维持登录状态，如果你在同一个浏览器中同时登录 Admin 和 CP Admin，后者的 Token 会覆盖前者。**必须使用 Chrome 的多个 Profile 或无痕模式进行跨角色测试。**

## 3. 高危测试领域 (High-Risk Areas)

在日常测试中，请将 80% 的精力集中在以下三个高危领域：

### 3.1 资金流与并发 (Fund Flow & Concurrency)
* **风险点**：CP 钱包被透支，或者发票被重复支付。
* **测试方法**：
  * **余额边界测试**：当钱包余额为 $100 时，尝试支付一张 $100.01 的发票，预期必须被拦截。
  * **并发点击测试**：使用工具（如 Postman 或脚本）对同一个“支付”接口同时发起 10 个请求。预期只有 1 个成功，其余 9 个必须返回失败，且钱包余额只扣减 1 次。

### 3.2 财务计算与精度 (Financial Calculations)
* **风险点**：汇率转换或百分比计算导致的精度丢失。
* **测试方法**：
  * **极端汇率测试**：测试某些面值极小的货币（如印尼盾 JPY 或越南盾 IDR），验证系统在除以汇率时，是否会出现无限循环小数导致的崩溃，或者四舍五入是否符合财务规范（通常是保留两位小数，采用银行家舍入法）。
  * **跨月折算测试**：测试员工在当月 28 日入职，或者在 3 日离职。验证系统按工作日比例折算（Proration）的基础薪水是否与手工用计算器算出来的一致。

### 3.3 数据隔离与越权 (Data Isolation)
* **风险点**：CP_A 的管理员看到了 CP_B 客户的薪资数据（这是严重的 GDPR 违规）。
* **测试方法**：
  * 抓包获取 CP_A 列表接口的请求参数（如 `customerId=10`）。
  * 登录 CP_B 的账号，使用 Postman 携带 CP_B 的 Token，强制请求 `customerId=10` 的数据。
  * 预期后端必须返回 `403 Forbidden` 或 `404 Not Found`，绝不能返回数据。

## 4. 测试工具与技巧

### 4.1 模拟系统时间 (Time Travel)
员工的状态流转（如 `Contract Signed` -> `Active`）和每月的发薪截账（Cut-off）严重依赖于系统时间。
* **技巧**：在本地测试环境中，不要傻等时间过去。你可以要求开发提供一个后门 API（或通过修改本地数据库的 `startDate` 字段），然后手动触发 Cron Job 的脚本，以验证时间敏感的逻辑。

### 4.2 拦截与修改第三方响应
系统集成了许多第三方服务（如发邮件、汇率 API）。
* **技巧**：在测试环境中，邮件通常会被拦截并输出到控制台（Console），而不是真的发出去。你需要查看后端终端的日志，验证白标邮件的发件人名称、Logo URL 是否正确拼接。

### 4.3 数据库验证 (Database Verification)
UI 上显示“支付成功”不代表底层数据是对的。
* **技巧**：安装一个 SQLite 客户端（如 DBeaver 或 VSCode 的 SQLite 插件）。每次执行完核心操作后，直接打开 `local.db`，检查 `walletTransactions`（流水表）和 `invoices`（发票表）的数据是否对齐。

## 5. 你的第一周任务

1. 搭建本地测试环境，并成功创建上述的“4 个角色测试账号”。
2. 按照 [测试场景清单](../07_测试/测试场景清单.md)，从头到尾走通一次“员工入职 -> 生成发票 -> CP 支付”的完整闭环。
3. 尝试制造一次“钱包余额不足”的报错。

在 Extend Global，QA 不是“点点点”的机器，而是守护资金安全的最后一道防线。保持对数据的敬畏之心！
