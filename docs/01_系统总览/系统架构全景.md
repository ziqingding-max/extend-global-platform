# 系统架构全景

本文档从宏观视角介绍 Extend Global (EG) 平台的技术架构、部署拓扑与数据流向，帮助产品、研发与运维团队建立对系统的全局认知。

## 1. 架构定位与设计哲学

Extend Global 平台采用现代化的前后端分离架构，核心设计哲学围绕“多租户隔离”与“白标定制（White-label）”展开。作为 B2B2B 的名义雇主（EOR）平台，系统不仅需要服务于 EG 内部运营团队，更需要赋能渠道合作伙伴（Channel Partner, CP），使其能够以自身品牌为终端客户提供服务。

为实现这一目标，系统在架构上将门户严格划分为三个独立的 SPA（单页应用），并通过通配符子域名路由机制，在同一套代码库中实现多品牌的动态渲染。后端则采用单一服务集群，通过数据层的外键关联实现租户隔离。

## 2. 核心技术栈

系统的技术选型侧重于类型安全、开发效率与运行稳定性。

| 层级 | 核心技术 | 描述与用途 |
| :--- | :--- | :--- |
| **前端框架** | React 18 + Vite | 提供高性能的渲染与快速的本地构建体验。 |
| **语言与类型** | TypeScript | 全栈采用 TypeScript，通过 tRPC 实现前后端端到端的类型安全。 |
| **UI 组件库** | TailwindCSS + Ant Design | Tailwind 负责灵活的原子化样式，Ant Design 提供复杂的交互组件。 |
| **后端框架** | Node.js + Express | 轻量级、高吞吐量的服务端运行环境。 |
| **API 层** | tRPC | 替代传统的 REST/GraphQL，无需手动编写接口文档，直接共享类型定义。 |
| **ORM 与数据库** | Drizzle ORM + SQLite | 采用基于文件的 SQLite（生产环境），配合 Drizzle 提供强类型的 SQL 查询能力。 |
| **文件存储** | 阿里云 OSS | 存储所有上传的文件、生成的 PDF 发票及白标 Logo，节点位于马来西亚。 |
| **部署与反向代理** | Docker Compose + Nginx | 容器化部署，Nginx 负责 SSL 终结与通配符子域名路由。 |

## 3. 门户划分与路由架构

系统由三个独立的前端应用组成，共享同一个后端 API 服务。Nginx 作为反向代理，通过解析请求的域名来决定渲染哪个门户。

### 3.1 门户分类

1. **Admin Portal (管理后台)**
   * **域名**: `admin.extendglobal.ai`
   * **受众**: EG 内部运营团队、财务人员、系统管理员。
   * **功能**: 平台全局配置、CP 管理、底价设置、跨 CP 财务报表查看、全局工单处理。

2. **App Portal (客户与CP门户)**
   * **域名**: `app.extendglobal.ai` (直客) 或 `*.extendglobal.ai` (CP白标客户)
   * **受众**: 渠道合作伙伴 (CP) 的管理员、终端企业客户 (Client) 的 HR/财务。
   * **功能**: CP 视角的客户管理、定价加价配置；终端客户视角的员工入职发起、账单支付、请假审批。

3. **Worker Portal (员工门户)**
   * **域名**: `worker.extendglobal.ai` (直客员工) 或 `*.extendglobal.ai/worker` (CP白标员工)
   * **受众**: 被雇佣的 EOR 员工、AOR 独立承包商。
   * **功能**: 个人信息维护、工资单查看、报销申请、请假申请。

### 3.2 通配符子域名路由机制

为实现 CP 白标功能，系统采用了通配符域名（`*.extendglobal.ai`）解析。当用户访问如 `acme.extendglobal.ai` 时，Nginx 会将请求代理至 Node.js 服务。前端 SPA 启动时，会解析当前 URL 的子域名，调用后端的 `/api/branding/:subdomain` 接口。

如果该子域名属于某个注册的 CP，前端会动态注入该 CP 的 Logo、主色调及公司名称，实现无缝的品牌替换。如果路径中包含 `/worker`，则渲染员工门户视图；否则默认渲染客户门户视图。

## 4. 部署拓扑结构

系统采用单机容器化部署方案，部署于阿里云马来西亚（吉隆坡）节点的 ECS 服务器上。

### 4.1 容器编排

通过 `docker-compose.prod.yml`，系统编排了三个核心容器：

1. **eg-saas-app**: 运行 Node.js 应用程序，包含前端静态文件服务与后端 API 接口。挂载本地目录 `/app/data` 用于持久化 SQLite 数据库文件。
2. **eg-saas-nginx**: 监听 80 与 443 端口，负责处理所有外部 HTTP/HTTPS 请求。挂载 Let's Encrypt 证书目录与 Nginx 配置文件。
3. **eg-saas-certbot**: 独立容器，负责定期（每12小时）检查并自动续期 SSL 证书。

### 4.2 数据流向

当外部请求到达服务器时，数据流向如下：

1. 用户浏览器发起 HTTPS 请求。
2. 请求到达 ECS 服务器的 Nginx 容器（端口 443）。
3. Nginx 终结 SSL 连接，并根据 `server_name` 匹配规则进行分发：
   * 静态资源请求（`.js`, `.css`, `.png` 等）直接由 Nginx 缓存或转发给 App 容器。
   * API 请求（以 `/api/` 开头）代理至 App 容器的 3000 端口。
   * SPA 页面请求（如 `/dashboard`）由 App 容器返回 `index.html`，随后由前端 React Router 接管路由。
4. App 容器中的 Node.js 进程处理业务逻辑，通过 Drizzle ORM 读写挂载在宿主机卷上的 SQLite 数据库文件（`production.db`）。
5. 涉及文件上传或下载时，Node.js 进程通过阿里云 SDK 与 OSS-AP-Southeast-3 节点进行通信。

## 5. 核心业务引擎分层

后端的业务逻辑并非简单堆砌在路由处理函数中，而是抽象为多个独立的业务引擎服务（位于 `server/services/` 目录下）。这种分层设计保证了复杂财务与合规逻辑的可测试性与可维护性。

1. **双层发票引擎 (Dual-Layer Invoice Engine)**: 负责在每次发薪周期后，基于原始成本数据，同时生成 Layer 1 (EG 对 CP) 和 Layer 2 (CP 对 Client) 的两份账单。
2. **资金流转引擎 (Fund Flow Engine)**: 跟踪并处理四方（EG, CP, Client, Worker）之间的资金流转，特别是 CP 钱包的充值与手动扣款逻辑。
3. **汇差剥离引擎 (FX Stripping Engine)**: 在多币种结算场景下，精确分离纯代收代付的成本（Pass-through）与通过汇率加价产生的利润（FX Markup）。
4. **双币种对账引擎 (Reconciliation Engine)**: 负责将面向客户的应收账单（结算币种）与面向供应商的应付账单（当地币种）进行自动匹配与差异计算。

通过上述架构设计，Extend Global 平台在保证系统轻量易部署的同时，完美支撑了复杂的 B2B2B 商业模式与严苛的财务合规要求。
