# 代码结构与分层 (Code Structure & Layering)

Extend Global (EG) 平台采用全栈 TypeScript 的 Monorepo 结构，通过 `tRPC` 实现了前后端类型安全共享。本文档详细解析项目的目录结构、各层的职责边界以及数据流向，是新加入的开发人员（或 AI Agent）阅读代码的地图。

## 1. 顶层目录结构

项目根目录 `/home/ubuntu/extend-global-platform-work` 下的核心结构如下：

| 目录/文件 | 核心职责 | 描述 |
| :--- | :--- | :--- |
| `client/` | 前端代码库 | React 18 SPA，包含所有三个门户（Admin, App, Worker）的页面与组件。 |
| `server/` | 后端代码库 | Node.js 服务，包含 tRPC 路由、业务服务引擎与数据库连接配置。 |
| `drizzle/` | 数据库定义 | Drizzle ORM 的 Schema 定义与迁移文件（Migrations）。 |
| `shared/` | 共享类型 | *(可选)* 存放前后端共享的通用类型或常量。目前大部分类型直接通过 tRPC 自动推导。 |
| `docs/` | 文档库 | 本交接文档体系的存放位置。 |
| `docker-compose.prod.yml` | 部署编排 | 生产环境的容器编排文件。 |
| `package.json` | 依赖管理 | 根目录的依赖与 npm scripts，管理整个 Monorepo 的构建与运行。 |

## 2. 后端架构分层 (Server)

后端的代码结构遵循典型的“控制器 - 服务 - 数据访问”三层架构，以确保业务逻辑的可测试性与复用性。所有后端代码位于 `server/` 目录下。

### 2.1 路由层 (Routers)
* **路径**：`server/routers/`
* **职责**：作为 API 的入口点，负责接收请求、校验输入参数（使用 Zod）、进行权限拦截（RBAC Middleware），然后将请求委托给服务层。
* **规则**：路由层**绝对不应**包含复杂的业务计算逻辑或直接编写复杂的 SQL 查询。
* **关键文件**：
  * `channelPartners.ts`: CP 相关的增删改查。
  * `invoiceGeneration.ts`: 触发发票生成的入口。
  * `fundFlow.ts`: 处理钱包充值与支付请求。

### 2.2 服务层 (Services)
* **路径**：`server/services/`
* **职责**：系统的“大脑”，包含所有核心业务逻辑、财务计算与第三方 API 调用。
* **规则**：服务层函数应该是无状态的（Stateless），高度可测试的。服务之间可以相互调用，但应避免循环依赖。
* **关键引擎**：
  * `dualLayerInvoiceService.ts`: 双层发票引擎。
  * `fundFlowEngine.ts`: 四方资金流转引擎。
  * `reconciliationEngine.ts`: 双币种对账引擎。
  * `fxStrippingEngine.ts`: 汇差剥离引擎。

### 2.3 数据访问层 (Data Access)
* **路径**：`server/services/db/` 或直接在 Service 中使用 `drizzle-orm`。
* **职责**：负责与 SQLite 数据库的直接交互，封装常用的查询以供上层调用。
* **设计特点**：由于使用了 Drizzle ORM，许多简单的查询直接写在了 Service 层中。但在处理复杂的跨表 Join 或聚合查询时，建议提取到专门的 db service 中。

## 3. 前端架构分层 (Client)

前端是一个庞大的 React SPA，通过 Vite 进行构建。由于系统包含三个不同的门户，前端在路由层面进行了严格的物理隔离。所有前端代码位于 `client/src/` 目录下。

### 3.1 页面级路由 (Pages)
* **路径**：`client/src/pages/`
* **架构特点**：页面按所属的门户被划分为不同的子目录，防止组件交叉污染。
  * `/pages/` (根目录): 属于 Admin Portal 的页面（如全局发票管理、CP 管理）。
  * `/pages/portal/` & `/pages/cp-portal/`: 属于 App Portal 的页面，分别对应直客视图和 CP 管理员视图。
  * `/pages/worker/`: 属于 Worker Portal 的页面（如工资单、报销申请）。

### 3.2 全局状态与 Hook
* **路径**：`client/src/hooks/`
* **关键 Hook**：`useCpBranding.ts`。这是实现白标机制的核心 Hook。它在应用初始化时解析当前 URL 的子域名，向后端请求对应的品牌配置，并将其注入到 React Context 中，供全局组件消费。

### 3.3 tRPC 客户端集成
* 前端不使用 `axios` 或 `fetch` 手动发送请求。所有 API 调用均通过 `trpc.react` 钩子完成。
* 例如：`const { data: invoices } = trpc.invoices.list.useQuery();`
* 这种方式保证了如果后端修改了接口的返回值类型，前端在编译阶段就会立即报错，彻底消除了接口文档不同步的问题。

## 4. 数据库层 (Drizzle)

由于系统强依赖于复杂的关系型数据（如计费规则、资金流水），数据库 Schema 的定义是整个项目的基石。

* **路径**：`drizzle/schema.ts`
* **技术选型**：生产环境使用本地文件数据库 SQLite (`production.db`)，通过 Drizzle ORM 进行操作。
* **设计规范**：
  * 所有的表定义必须导出为常量（如 `export const invoices = sqliteTable(...)`）。
  * 必须使用 Zod 配合 Drizzle 生成类型定义（如 `export type Invoice = typeof invoices.$inferSelect`），以供 Service 层使用。
  * 所有的金额字段（如 `amount`, `balance`）在数据库中存储为 `real`（浮点数）或 `integer`（以分为单位的整数，取决于具体实现），并在代码中进行精度控制。

## 5. 核心数据流向示例

以“CP 手动支付 Layer 1 发票”为例，完整的数据流向如下：

1. **前端交互**：CP 管理员在 `client/src/pages/cp-portal/CpPortalWallet.tsx` 页面点击“支付”按钮。
2. **发起请求**：前端调用 `trpc.fundFlow.payLayer1Invoice.useMutation()`。
3. **路由拦截**：请求到达 `server/routers/fundFlow.ts`，tRPC 中间件验证用户是否拥有 CP Admin 权限，并提取 JWT 中的 `channelPartnerId`。
4. **服务调用**：路由层将参数传递给 `server/services/fundFlowEngine.ts` 的 `payLayer1FromWallet` 函数。
5. **事务执行**：
   * 引擎开启 Drizzle 数据库事务。
   * 查询 `channelPartnerWallets`，校验余额。
   * 扣减余额，在 `walletTransactions` 插入流水。
   * 更新 `invoices` 表中的状态为 `PAID`。
   * 提交事务。
6. **响应返回**：引擎返回成功结果，路由层将其透传给前端，前端更新 UI 状态。

理解这种自上而下、类型安全的数据流，是高效进行二次开发的关键。
