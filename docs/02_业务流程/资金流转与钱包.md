# 资金流转与钱包体系 (Fund Flow & Wallet)

在 Extend Global (EG) 的 B2B2B 模式下，资金的流转是最为敏感和核心的业务环节。为了在合规性、操作便利性与平台风险控制之间取得平衡，系统设计了一套基于“CP 手动钱包”的四方资金流转机制。本文档详细解析该机制的设计逻辑与操作流程。

## 1. 核心设计原则：拒绝自动扣款

在设计资金流转引擎时，系统遵循一个不可逾越的硬性原则：**系统绝对不执行未经 CP 授权的自动扣款。**

### 1.1 为什么不自动扣款？
在早期的系统设计中（Batch G 之前），曾设想当终端客户（Client）支付了 Layer 2 账单后，系统自动从 CP 的钱包中扣除 Layer 1 的成本。但在实际合规审查中，这种做法存在严重缺陷：
1. **资金错配风险**：终端客户的款项是打入 CP 的线下银行账户的，系统无法实时验证这笔款项是否真正到账。如果系统根据前端的“已支付”标记就自动扣除 CP 钱包的真实余额，极易引发 CP 的资金链断裂。
2. **汇率波动争议**：跨国转账存在时间差，自动扣款可能在汇率不利的时间点发生，引发财务纠纷。
3. **法律责任边界**：作为技术平台，EG 无权在没有明确授权指令的情况下，擅自动用属于 CP 的预充值资金。

因此，当前的资金流转完全基于 **CP 的主动触发与确认**。

## 2. 四方资金流转全景

完整的资金流转链路分为“客户付款给 CP”和“CP 付款给 EG”两个独立但关联的阶段。

### 2.1 阶段一：Client → CP (线下转账 + 线上标记)
1. **账单生成**：系统生成 Layer 2 发票，包含 CP 的加价。
2. **客户付款**：Client 收到发票后，通过传统的银行电汇（Wire Transfer）或第三方支付工具，将款项直接汇入 **CP 自己的银行账户**。
3. **状态标记**：CP 的财务人员在查收银行流水确认到账后，登录 CP Portal，找到对应的 Layer 2 发票，手动点击 **"Mark as Paid"**。
   * *注意：这一步纯粹是状态流转，系统内部不发生任何真实的资金转移。*

### 2.2 阶段二：CP → EG (钱包预充值 + 手动扣款)
EG 需要收取底层服务费和代发薪资（Layer 1 发票）。这部分资金通过 CP Wallet（渠道钱包）进行流转。

1. **钱包充值 (Top-up)**：
   * CP 预先将资金（如 $50,000）通过银行电汇打入 EG 的官方银行账户。
   * EG 财务人员在 Admin Portal 确认资金到账后，为该 CP 的虚拟钱包增加对应余额。
2. **发起支付**：
   * 当 Layer 1 发票生成后，CP 登录门户进入 "My Wallet" 页面。
   * CP 勾选需要支付的 Layer 1 发票，点击 **"Pay from Wallet"**。
3. **系统扣款与核销**：
   * `fundFlowEngine` 接收到支付指令，校验钱包余额是否充足。
   * 如果余额充足，系统在 `channelPartnerWallets` 中扣减相应金额，记录一条类型为 `deduction` 的流水。
   * 系统将对应的 Layer 1 发票状态更新为 `PAID`。
4. **最终下发**：
   * EG 确认 Layer 1 账单已结清后，才会将最终的净薪资发放给当地员工（Worker）或 Vendor。

## 3. 钱包系统的数据结构

钱包系统通过严密的事务控制确保资金的绝对安全。

### 3.1 核心表结构
* `channelPartnerWallets`：记录每个 CP 在不同币种下的当前余额（`balance`）。每个 CP 针对每种结算货币（如 USD, EUR）拥有一个独立的子钱包。
* `walletTransactions`：记录每一笔资金变动的流水账（Ledger）。包含以下类型：
  * `deposit`：充值增加余额。
  * `deduction`：支付发票扣减余额。
  * `refund`：发生错误或作废发票时的退款。

### 3.2 事务与并发控制
当 CP 发起 "Pay from Wallet" 操作时，系统通过 Drizzle ORM 开启数据库事务（Transaction）。
1. 锁定钱包记录（`SELECT ... FOR UPDATE`），防止并发扣款导致余额透支。
2. 检查余额 `>=` 发票金额。
3. 执行扣减并写入流水。
4. 更新发票状态。
5. 提交事务。如果任何一步失败，事务回滚，确保资金数据的一致性。

## 4. 保证金与冻结机制 (Frozen Wallet)

为了降低 EG 的资金垫付风险，系统引入了保证金（Deposit）机制。

1. **保证金收取**：当 CP 签约新客户或客户新增员工时，系统会生成一份特殊的 `Deposit Invoice`（通常为员工1-2个月的薪资预估）。
2. **资金冻结**：CP 支付保证金账单后，这笔资金进入 `channelPartnerFrozenWallets`（冻结钱包），而非可用余额。
3. **资金解冻**：只有当员工离职（Offboarding）且所有最终账单结清后，对应的保证金才会从冻结钱包转移回可用钱包，供 CP 提现或抵扣后续账单。

通过这套基于手动确认与预充值钱包的机制，Extend Global 平台有效隔离了 B2B2B 模式下的资金风险，同时为 CP 提供了清晰的资金流转视图（Margin）核算看板。
