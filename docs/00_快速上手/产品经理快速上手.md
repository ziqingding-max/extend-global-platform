# 产品经理 (PM) 快速上手指南

欢迎加入 Extend Global (EG) 团队！作为 B2B2B EOR 平台的产品经理，你面临的挑战不仅仅是设计好用的界面，更是如何平衡“合规、资金安全与多层级用户体验”。

本指南旨在帮助你在 3 小时内建立起对整个系统的产品全局观。

## 1. 必须理解的核心概念

在开始画原型或写 PRD 之前，请务必先阅读以下三篇核心文档：
1. **[四方角色与流转](../02_业务流程/四方角色与流转.md)**：这是整个系统的灵魂。你必须时刻清楚，系统中有四个角色（EG, CP, Client, Worker），你在设计的每一个功能，到底是在为哪个角色服务？对其他角色有什么影响？
2. **[资金流转与钱包](../02_业务流程/资金流转与钱包.md)**：为什么我们不自动扣款？为什么会有“冻结钱包”？理解资金流，你才能理解为什么某些看似繁琐的确认步骤是不可省略的。
3. **[双层发票引擎](../02_业务流程/双层发票引擎.md)**：这是 CP 能够赚钱的基础。你需要理解底价（Layer 1）和对客价（Layer 2）的区别。

## 2. 产品的三个“平行宇宙” (Portals)

EG 平台在物理上是一个代码库，但在产品呈现上，它被硬性切割成了三个完全独立的门户（Portal）。作为 PM，你需要学会在三个宇宙中切换视角。

### 2.1 Admin Portal (上帝视角)
* **使用者**：EG 内部的运营、财务、法务团队。
* **核心功能**：审核入职、配置底层定价、生成全局发票、处理跨国打款、查看净额法 P&L。
* **设计原则**：**效率与准确性优先**。这里的用户是专业人士，他们需要的是强大的批量处理能力、密集的数据表格和不可篡改的审计日志。

### 2.2 App Portal (白标宇宙)
* **使用者**：CP 管理员 和 终端客户 (Client Admin)。
* **核心功能**：CP 配置对客加价、充值钱包；Client 发起入职、审批请假、支付账单。
* **设计原则**：**白标与隔离**。这里的所有界面（Logo、颜色、发件人邮箱）都必须是动态的。你不能在这个门户中硬编码任何“Extend Global”的字样。同时，必须绝对保证 CP A 看不到 CP B 的数据。

### 2.3 Worker Portal (员工宇宙)
* **使用者**：被雇佣的全球员工或承包商。
* **核心功能**：签署合同、查看工资单、申请请假和报销。
* **设计原则**：**简单与合规**。员工可能不熟悉复杂的系统，界面必须极度傻瓜化。同时，根据员工所在国家的不同，展示的字段必须动态变化（如美国员工需要填 W-4，而香港员工不需要）。

## 3. 功能配置项清单 (What is Configurable?)

在接手系统时，PM 经常会问：“这个规则是写死的，还是可以配置的？”以下是目前系统支持的核心配置项，了解这些可以避免你提出需要重构底层代码的“伪需求”：

| 配置模块 | 颗粒度 | 谁来配置 | 业务影响 |
| :--- | :--- | :--- | :--- |
| **白标配置 (Branding)** | CP 级 | Admin / CP Admin | 决定 App/Worker Portal 的 Logo、主题色和子域名。 |
| **底层定价 (Layer 1)** | CP 级 / 客户级 | 仅限 Admin | 决定 EG 向 CP 收取多少钱（支持固定、比例、阶梯）。 |
| **对客定价 (Layer 2)** | 客户级 | CP Admin | 决定 CP 向终端客户收取多少钱（决定 CP 的利润）。 |
| **国家指南 (Country Guide)** | 国家级 | Admin | 决定该国的法定社保费率、封顶线、法定年假天数。 |
| **客户休假政策 (Leave Policy)** | 客户级 | Client Admin | 可以在国家法定底线之上，为员工配置额外的带薪假。 |
| **汇率加价 (FX Markup)** | 全局 / CP 级 | Admin | 决定系统展示给客户的汇率比真实汇率高多少。 |

## 4. 常见的产品设计陷阱 (Pitfalls)

在设计新功能时，请警惕以下陷阱：

1. **忽视跨国时区**：在设计“请假”或“截账日”功能时，必须考虑时区问题。员工在东京点击“今天请假”，系统后台记录的日期在纽约可能还是“昨天”。所有的日期存储必须使用 UTC。
2. **想当然地“自动执行”**：在涉及资金、解雇、合同修改的场景，不要轻易设计“系统自动执行”的逻辑。在欧洲，非法解雇会导致巨额罚款，必须留有法务人工介入的“断点”。
3. **忘记了 Vendor 的存在**：在设计发薪流程时，不要以为只要客户付了钱，员工就能立刻收到钱。中间通常还隔着一个当地的落地服务商（Vendor）。设计时要考虑 Vendor 处理资金的 T+2 延迟。

## 5. 你的第一周行动建议

1. **走一遍主流程**：向开发要一个测试环境的账号。扮演一次 Client 发起入职，扮演一次 Admin 审核并生成发票，扮演一次 CP 支付钱包。
2. **查看一次 P&L**：理解总额（Gross）和净额（Net）的区别，这是 EG 商业模式的核心。
3. **阅读合规手册**：至少浏览一遍 [欧洲合规手册](../04_合规边界手册/欧洲合规手册.md)，了解 GDPR 和社保封顶线对产品设计带来的硬性约束。

祝你在 Extend Global 的产品之旅顺利！
