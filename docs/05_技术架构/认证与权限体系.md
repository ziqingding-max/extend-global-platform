# 认证与权限体系 (Authentication & RBAC)

Extend Global (EG) 平台作为一个支持多租户（CP）、多层级角色的 B2B2B SaaS 系统，其数据安全性与操作合规性高度依赖于底层的认证与权限控制机制。

本文档详细解析系统的身份认证流程（Authentication）、基于角色的访问控制（RBAC）设计，以及如何在 tRPC 路由层实现硬性的数据隔离。

## 1. 身份认证机制 (Authentication)

系统采用基于 JWT (JSON Web Token) 的无状态认证机制。所有前端门户（Admin, App, Worker）均共享同一套后端的认证逻辑，但登录入口和成功后的路由重定向有所不同。

### 1.1 JWT 载荷结构 (Payload)
当用户成功登录后，后端会签发一个 JWT。该 Token 的 Payload 中包含以下关键信息，这些信息是后续权限校验的基石：
* `userId`: 用户的唯一主键 ID。
* `role`: 用户的全局角色（如 `admin`, `cp_admin`, `client_admin`, `worker`）。
* `channelPartnerId`: 如果用户属于某个 CP，则包含此字段（对于 Admin 为 null）。
* `customerId`: 如果用户属于某个终端客户，则包含此字段。

### 1.2 跨域与 Cookie 策略
* Token 通过 HTTP-Only 的 Cookie 下发，以防止 XSS 攻击窃取 Token。
* 考虑到 CP 白标模式下存在跨子域名请求（如 `acme.extendglobal.ai` 请求 API），Cookie 的 `Domain` 属性配置为顶层域名（`.extendglobal.ai`），确保在不同子域名间共享登录状态。
* `SameSite` 属性在生产环境中通常设置为 `Lax` 或 `None`（配合 `Secure=true`），以支持跨站点的 API 调用。

### 1.3 密码重置与安全
* 系统不存储明文密码，所有密码均通过 `bcrypt` 进行加盐哈希处理。
* 密码重置流程：用户请求重置 → 后端生成一个短期有效的随机 Token（存储在数据库中并附带过期时间） → 发送包含重置链接的白标邮件 → 用户点击链接输入新密码。

## 2. 角色与权限矩阵 (RBAC)

系统内置了五种核心角色，每种角色对应着严格的操作边界。

| 角色 (Role) | 登录门户 | 核心权限与可见性边界 |
| :--- | :--- | :--- |
| **Admin** (系统管理员) | Admin Portal | **上帝视角**。可以查看和修改所有 CP、所有客户的数据。负责配置底层定价、审核发票、处理跨国资金下发。 |
| **CP Admin** (渠道管理员) | App Portal (CP 视图) | **CP 级隔离**。只能查看归属于自己 `channelPartnerId` 的客户和员工数据。负责配置对客加价、向客户发送发票、从钱包支付底层成本。 |
| **Client Admin** (客户管理员) | App Portal (客户视图) | **客户级隔离**。只能查看归属于自己 `customerId` 的员工数据。负责发起入职、审批请假报销、支付最终发票。 |
| **Worker** (员工/承包商) | Worker Portal | **个人级隔离**。只能查看自己的个人档案、工资单和合同。 |
| **Vendor** (落地服务商) | Admin Portal (受限视图) | *(预留扩展)* 仅能查看指派给该 Vendor 的员工数据，上传实际的 Vendor Bill。 |

## 3. tRPC 路由层的权限拦截

在 Extend Global 平台中，权限控制并非在前端隐藏几个按钮那么简单，而是必须在后端的 API 层（tRPC Routers）进行硬性拦截。

### 3.1 tRPC 中间件设计
系统通过定义不同的 tRPC Procedure（过程）来实现权限的声明式拦截。在 `server/routers/_core/procedures.ts` 中，定义了以下基础 Procedure：

* `publicProcedure`: 无需登录即可访问的接口（如登录、注册、获取白标配置）。
* `protectedProcedure`: 必须携带有效 JWT 才能访问的接口。
* `adminProcedure`: 仅限 `role === 'admin'` 访问。
* `cpAdminProcedure`: 仅限 `role === 'admin'` 或 `role === 'cp_admin'` 访问。

### 3.2 数据隔离查询 (Data Isolation)
仅仅限制接口的访问权限是不够的，系统还必须在 SQL 查询层面注入隔离条件，防止“越权查询（IDOR）”。

**错误示例（存在越权风险）**：
```typescript
// 如果 CP Admin 传入了一个不属于他的 customerId，这段代码会返回别人的数据！
const customers = await db.select().from(customers).where(eq(customers.id, input.customerId));
```

**正确示例（强制数据隔离）**：
```typescript
// 在 cpAdminProcedure 中
export const getCustomer = cpAdminProcedure
  .input(z.object({ customerId: z.number() }))
  .query(async ({ ctx, input }) => {
    // 提取当前登录用户的上下文
    const { user } = ctx;
    
    // 构建基础查询
    let query = db.select().from(customers).where(eq(customers.id, input.customerId));
    
    // 如果不是 Admin，强制附加 CP 隔离条件
    if (user.role !== 'admin') {
      query = query.where(and(
        eq(customers.id, input.customerId),
        eq(customers.channelPartnerId, user.channelPartnerId) // 强制校验归属权
      ));
    }
    
    const result = await query;
    if (!result.length) throw new TRPCError({ code: 'NOT_FOUND' });
    return result[0];
  });
```

### 3.3 跨租户操作的严禁原则
在开发涉及资金、发票或状态流转的接口时，开发人员必须遵循一个硬性原则：**永远不要信任前端传入的关联 ID，必须通过当前登录用户的 Context 进行二次校验。**

例如，当 CP 发起钱包扣款时，后端不能仅仅依赖前端传来的 `walletId`，而是必须通过 `ctx.user.channelPartnerId` 去数据库中反查该钱包是否真的属于该 CP，否则可能导致一个 CP 恶意扣减另一个 CP 的钱包余额。

## 4. 审计与合规监控

为了满足金融级 SaaS 的合规要求，所有高危操作（特别是 Admin 角色执行的操作）都会被记录在 `auditLogs` 表中。

* **记录范围**：包括但不限于修改定价规则、手动修改发票状态、人工确认钱包充值、强制修改员工状态等。
* **记录内容**：操作人 ID、操作时间、操作类型（Action Type）、涉及的实体 ID、以及修改前后的数据快照（JSON 格式）。
* **用途**：当发生财务纠纷或系统异常时，审计日志是追溯责任的唯一可信凭证。开发人员在新增敏感业务逻辑时，必须同步调用 `auditLogService` 写入日志。
