# 核心测试场景清单 (Core Testing Scenarios)

Extend Global (EG) 平台作为一个涉及跨境资金流转和复杂计费规则的金融级 HR SaaS，其测试的严谨性直接关系到公司的存亡。本文档为 QA 团队和开发人员提供了一套核心的端到端（E2E）测试场景清单，这些场景在每次大版本发布（发版）前**必须**被回归验证。

## 1. 测试策略与原则

1. **以资金流为核心**：UI 样式或文案的 Bug 是次要的，但任何导致发票金额计算错误、钱包余额透支或越权访问的 Bug 都是 **P0 级致命缺陷**。
2. **多角色协同测试**：绝大多数核心流程都需要跨门户、跨角色完成。测试用例必须包含 Admin、CP Admin、Client Admin 和 Worker 的角色切换。
3. **数据隔离验证**：在每个测试场景中，都必须尝试使用非授权账号（如 CP_A 尝试访问 CP_B 的数据）来验证后端的 RBAC 和数据隔离是否生效。

---

## 2. 核心场景 1：白标入职与生命周期流转

**目标**：验证员工能否顺利走完从被邀请到正式在职的流程，且整个过程中的品牌展示（Logo、域名、邮件）完全符合 CP 的白标配置。

| 步骤 | 操作角色 | 操作内容 | 预期结果 (验证点) |
| :--- | :--- | :--- | :--- |
| 1.1 | Client Admin | 在 App Portal 发起新员工入职邀请（选择国家、填写薪资）。 | 系统生成 `Pending Review` 状态的记录。 |
| 1.2 | Admin | 在 Admin Portal 审核通过该入职工单。 | 员工状态变为 `Documents Incomplete`。系统向员工发送邀请邮件。**【关键验证】邮件发件人名称、Logo 必须是该客户所属 CP 的品牌，而非 EG。** |
| 1.3 | Worker | 点击邮件链接，访问 Worker Portal，填写个资并上传护照。 | **【关键验证】浏览器 URL 必须是 CP 的子域名（如 hr.cp-domain.com），页面主题色和 Logo 必须匹配。** 提交后状态变为 `Onboarding`。 |
| 1.4 | Admin | 在 Admin Portal 上传签署好的劳动合同 PDF，并标记为已双签。 | 员工状态变为 `Contract Signed`。 |
| 1.5 | System Cron | 模拟系统时间推进到员工的 `startDate`。 | 触发 Cron Job 后，员工状态自动变为 `Active`。**【关键验证】系统自动为员工分配当年的法定年假额度。** |

## 3. 核心场景 2：发薪计算与双层发票生成

**目标**：验证薪酬引擎能否正确处理中途入职的折算（Proration），以及定价引擎能否正确生成包含 CP 加价的双层发票。

**前置条件**：
* 存在一个 `Active` 状态的员工，月薪 $5,000，当月 15 日入职。
* CP 的 Layer 1 底价配置为：按比例 5%。
* 客户的 Layer 2 对客价配置为：固定 $500。

| 步骤 | 操作角色 | 操作内容 | 预期结果 (验证点) |
| :--- | :--- | :--- | :--- |
| 2.1 | Admin | 触发该员工所在国家的当月 Payroll Run 计算。 | **【关键验证】基础薪资必须按工作日比例折算（如 $2,500）。** 自动计算出雇主社保和个税扣款。 |
| 2.2 | Admin | 确认 Payroll Run，触发 `Generate Invoices`。 | 系统生成两张草稿发票：Layer 1 和 Layer 2。 |
| 2.3 | Admin | 检查 Layer 1 发票金额。 | **【关键验证】服务费金额 = (折算后薪资 + 雇主社保) * 5%。** |
| 2.4 | Admin | 检查 Layer 2 发票金额。 | **【关键验证】服务费金额 = 固定 $500。** 代收代付成本（薪水+社保）必须与 Layer 1 完全一致。 |
| 2.5 | Admin | 点击 `Publish Invoices`。 | 发票状态变为 `Published`。系统向 CP 和 Client 发送账单邮件。 |

## 4. 核心场景 3：四方资金流转与并发扣款防御

**目标**：验证最核心的资金流转引擎，特别是当 CP 钱包余额不足或发生并发点击时的系统防御机制。

**前置条件**：
* CP 拥有两张待支付的 Layer 1 发票：Invoice_A ($10,000) 和 Invoice_B ($5,000)。
* CP 的当前可用钱包余额为：$12,000。

| 步骤 | 操作角色 | 操作内容 | 预期结果 (验证点) |
| :--- | :--- | :--- | :--- |
| 3.1 | Client Admin | 登录 App Portal，查看 Layer 2 发票，点击“标记为已线下付款”。 | 发票状态变为 `Paid`。*(注意：此操作不影响 CP 钱包余额)* |
| 3.2 | CP Admin | 登录 App Portal，进入钱包页面，勾选 Invoice_A 和 Invoice_B，尝试批量支付。 | **【关键验证】系统必须拦截该请求，提示“余额不足”，发票状态不改变，钱包余额不扣减。** |
| 3.3 | CP Admin | 仅勾选 Invoice_A，使用自动化脚本模拟**同时发起 5 个并发的支付请求**。 | **【关键验证】数据库事务锁（FOR UPDATE）必须生效。只有 1 个请求成功，扣减 $10,000，其余 4 个请求失败。最终钱包余额必须是 $2,000，绝不能变成负数。** |
| 3.4 | CP Admin | 检查钱包流水 (Ledger)。 | 新增一条类型为 `deduction`、金额为 $10,000 的流水，关联 Invoice_A。 |
| 3.5 | Admin | 在 Admin Portal 检查 Invoice_A 的状态。 | 状态变为 `Paid`。此时允许财务团队向该发票包含的员工执行最终的资金下发。 |

## 5. 核心场景 4：双币种对账与汇差剥离

**目标**：验证财务对账引擎能否准确识别成本变动，并将汇率波动带来的损益正确计入平台 P&L。

**前置条件**：
* 月中生成的 Invoice 包含一名日本员工：预估成本 150,000 JPY。当时系统锁定汇率 1 USD = 150 JPY（预估成本 $1,000）。

| 步骤 | 操作角色 | 操作内容 | 预期结果 (验证点) |
| :--- | :--- | :--- | :--- |
| 4.1 | Admin | 月底收到 Vendor Bill，上传至系统。账单显示实际成本变为 155,000 JPY（增加了 5,000 JPY 的临时体检费）。 | 系统解析账单，自动与该员工的 Invoice 进行匹配。 |
| 4.2 | Admin | 输入 Vendor 实际打款当天的真实汇率：1 USD = 140 JPY。 | 引擎开始计算差异（Variance）。 |
| 4.3 | Admin | 查看对账结果面板。 | **【关键验证】引擎必须精准拆解出：**<br>1. **Local Variance**: 5,000 JPY。这部分被标记为需向客户追收的 Adjustment。<br>2. **FX Variance**: 由于汇率从 150 跌到 140，导致平台产生汇兑损失。该损失由平台承担，不向客户追收。 |
| 4.4 | Admin | 查看当月的 Net P&L 报表。 | **【关键验证】总额法下的巨额代收代付资金必须被剥离。净收入仅显示收取的服务费，并扣减了步骤 4.3 中计算出的汇兑损失。** |

## 6. 异常边界与合规阻断测试

除了上述正常主流程，还必须验证系统在非法操作下的“阻断”能力：

* **越权访问阻断**：使用 Worker 的 Token 尝试调用 `trpc.invoices.list` 接口，预期返回 `403 Forbidden`。
* **非法解雇阻断**：Client Admin 尝试在系统内直接设定一名德国员工的“明天离职”，预期系统弹窗拦截，提示“欧洲国家需经过法务审核，已生成工单”，不允许客户单方面设定离职日。
* **定价倒挂阻断**：CP Admin 尝试将 Layer 2 对客价修改得比 Layer 1 底价还要低，预期保存失败，提示“Margin cannot be negative”。

通过在每次发版前严格执行上述场景，QA 团队可以最大限度地保障 Extend Global 平台在高速迭代中的财务准确性与合规安全性。
