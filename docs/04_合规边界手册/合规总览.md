# 合规总览 (Compliance Overview)

本文档是 Extend Global (EG) 平台合规体系的最高指导文件，专为注册会计师 (CPA)、审计师、税务师及内部法务团队编写。它界定了平台在 B2B2B 商业模型下的法律定位、财务核算原则以及数据隐私保护边界。

## 1. 业务定性与法律实体关系

在评估 EG 平台的合规风险时，首先必须明确各方的法律关系。EG 平台本身**不提供**直接的劳务派遣，而是提供一套“合规雇佣基础设施”。

### 1.1 核心法律关系 (Legal Relationships)
1. **EG 与 渠道合作伙伴 (CP)**：签署《平台软件使用及渠道分销协议 (SaaS & Channel Distribution Agreement)》。EG 授权 CP 使用白标系统，并承诺提供底层的 EOR 履约服务。
2. **CP 与 终端客户 (Client)**：签署《名义雇主服务主协议 (Master Service Agreement, MSA)》。在此协议中，CP 是面向客户的唯一服务提供商，EG 不作为协议主体出现。
3. **EG (或当地 Vendor) 与 员工 (Worker)**：签署受当地劳动法管辖的《劳动合同 (Employment Contract)》。在法律意义上，EG（或其指定的当地实体）是员工的唯一合法雇主。

### 1.2 B2B2B 模式下的责任隔离
这种结构在法律上构建了一道防火墙：
* 终端客户如果因为业务原因需要解雇员工，必须通过 CP 向 EG 发出指令，由 EG 根据当地劳动法执行解雇程序。客户不能直接解雇员工。
* 如果发生劳动纠纷，员工起诉的对象是其名义雇主（EG 的当地实体），而非终端客户或 CP。EG 负责出庭应诉，但相关赔偿成本（如果是由于客户的非法指令导致）将通过合同条款向客户追偿。

## 2. 财务合规与审计原则

EG 的财务系统设计严格遵循国际财务报告准则 (IFRS) 与美国通用会计准则 (US GAAP)，特别是关于收入确认与代理人/主要责任人的判定。

### 2.1 收入确认：净额法 (Net-basis Reporting)
如 [净额法 P&L 报表](../03_财务引擎/净额法P&L报表.md) 中所述，EG 在代收代发薪资的过程中，不承担存货风险（即如果不发薪，EG 不会蒙受除服务费外的损失），且对薪资定价没有自主决定权（由客户决定）。

因此，根据 **IFRS 15 / ASC 606** 的指导原则，EG 在此交易中扮演的是**代理人 (Agent)** 角色。
* **合规要求**：代收的薪资与税费必须作为资产负债表上的“应付账款/代收代付款”处理，绝对不能计入利润表的“营业收入 (Revenue)”。
* **审计关注点**：审计师应重点检查系统生成的 `Net P&L Report`，验证代收代付资金的剥离逻辑是否准确，是否存在虚增收入的情况。

### 2.2 资金隔离与反洗钱 (AML)
由于平台涉及巨额的跨境资金流动，反洗钱是合规的重中之重。
* **KYC / KYB**：所有 CP 和直接签约的终端客户在激活账户前，必须通过企业实名认证 (KYB)，提供营业执照、董事信息及最终受益人 (UBO) 声明。
* **资金原路返回原则**：如果发生退款，资金必须原路退回至付款的银行账户，严禁退入第三方账户。
* **CP 钱包资金属性**：CP 钱包中的余额属于“预付款 (Prepaid Expenses)”，在未被用于支付 Layer 1 发票前，EG 不能将其确认为收入，必须存放在专门的隔离银行账户中。

## 3. 数据隐私与安全合规

作为一个处理全球员工敏感个人信息（如护照、银行账户、薪资明细）的平台，EG 必须遵守全球最严格的数据保护法规。

### 3.1 GDPR (欧盟) 与类似法规
* **数据主体权利**：系统支持员工行使“被遗忘权 (Right to be Forgotten)”。当员工离职并度过法定追溯期后，Admin 可以执行硬删除或匿名化处理。
* **数据最小化**：系统在设计表结构时，严格区分了“业务必须数据”与“可选数据”。对于不需要的敏感信息（如宗教信仰、种族），系统拒绝收集。
* **白标环境下的数据处理者身份**：在 CP 模式下，终端客户是“数据控制者 (Data Controller)”，CP 是“一级数据处理者 (Primary Data Processor)”，EG 则是“子处理者 (Sub-processor)”。EG 仅根据系统的预设逻辑处理数据，不进行任何商业化的数据挖掘。

### 3.2 基础设施安全
* **数据加密**：所有静态数据（特别是存储在 SQLite 中的敏感字段，如 `bankAccountNumber`）必须在应用层或数据库层进行加密（AES-256）。传输中的数据强制使用 TLS 1.2+。
* **访问控制**：实行严格的 RBAC（基于角色的访问控制）。开发人员无法访问生产环境的真实数据，运维人员的数据库访问记录受审计日志（Audit Logs）监控。

## 4. 各国特定的合规要求

虽然本总览提供了宏观的合规框架，但 EOR 业务的本质是“极度本地化”的。每个国家在劳动法、个税申报、社保缴纳上都有独特的规定。

为此，我们针对核心业务区域编写了专门的合规手册：
* 针对亚太区金融中心与跳板：请参阅 [香港合规手册](./香港合规手册.md)。
* 针对数据隐私与劳工保护最严格的区域：请参阅 [欧洲合规手册](./欧洲合规手册.md)。

审计师与法务团队在进行尽职调查时，应结合本总览与具体的区域手册，全面评估 Extend Global 平台的合规健康度。
