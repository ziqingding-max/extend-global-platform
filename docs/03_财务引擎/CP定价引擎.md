# CP 定价引擎 (CP Pricing Engine)

在 Extend Global (EG) 的 B2B2B 商业模型中，灵活且多层级的定价策略是吸引渠道合作伙伴 (CP) 的核心武器。系统通过 `cpPricingEngine`（CP定价引擎）来管理和计算这些复杂的收费规则。

本文档详细介绍定价引擎的架构设计、规则配置方式以及在双层发票中的计算应用。

## 1. 定价架构的双层设计

与双层发票相对应，系统的定价体系也分为两个层级，分别对应不同的数据库表和配置权限。

### 1.1 Layer 1 定价 (EG ↔ CP)
* **业务含义**：EG 向 CP 收取的“底价（Bottom Price）”。
* **配置权限**：仅限 EG 内部运营人员（Admin）配置。
* **数据存储**：`cpPricingRules` 表。
* **颗粒度**：通常配置在 CP 级别（即该 CP 旗下的所有客户默认适用此底价），但也支持针对该 CP 旗下的特定客户配置特殊底价（覆盖全局配置）。

### 1.2 Layer 2 定价 (CP ↔ Client)
* **业务含义**：CP 向终端客户收取的“对客价（Markup Price）”。
* **配置权限**：CP 的财务或管理员可以在其白标门户（App Portal）中自主配置。
* **数据存储**：`customerPricing` 表（记录终端客户的特定协议价）或 CP 自身的默认加价策略。
* **颗粒度**：通常配置在终端客户（Client）级别，允许 CP 对不同的大客户或小客户采用不同的加价策略。

## 2. 支持的定价模型 (Pricing Models)

定价引擎支持多种灵活的计费模型，以适应不同国家、不同用工类型的商业需求。在数据库中，这通过 `pricingType` 字段来区分。

### 2.1 固定人头费 (Fixed per Head)
* **说明**：无论员工薪资高低，每月收取固定的服务费。
* **配置示例**：`feeAmount = 150`，`currency = USD`。
* **适用场景**：大多数发达国家的标准 EOR 业务。

### 2.2 按比例收费 (Percentage of Payroll)
* **说明**：服务费按员工当月总用工成本（Gross Pay + Employer Contributions）的百分比收取。
* **配置示例**：`feePercentage = 5.0` (即 5%)。
* **适用场景**：某些高风险或高管理成本的新兴市场国家。

### 2.3 阶梯定价 (Tiered Pricing)
* **说明**：根据该客户（或该 CP）在职员工的总人数，动态决定适用的人头费。人数越多，单价越低。
* **配置示例**：存储在 `tierRules` JSON 字段中。
  * Tier 1: 1-10 人，每人 $150
  * Tier 2: 11-50 人，每人 $130
  * Tier 3: 51+ 人，每人 $110
* **适用场景**：鼓励 CP 或大客户扩大用工规模的促销策略。

### 2.4 最低消费 (Minimum Fee)
* **说明**：通常与“按比例收费”结合使用。如果按比例计算出的服务费过低，则按最低消费金额收取，以覆盖平台的基础运营成本。
* **配置示例**：`feePercentage = 5.0`，`minFee = 100`，`currency = USD`。如果某员工总成本为 $1000，5% 是 $50，低于 $100，则该月服务费按 $100 收取。

## 3. 引擎的计算解析流程

当发薪周期结束，系统调用 `dualLayerInvoiceService` 生成发票时，定价引擎会介入并执行以下计算流：

### 3.1 步骤 1：确定计费基数
引擎首先统计该终端客户在当月处于 `Active` 或 `On Leave` 状态的员工数量（用于阶梯定价判断），并汇总每位员工的当月总用工成本（用于按比例收费计算）。

### 3.2 步骤 2：解析 Layer 1 底价
1. 引擎查询 `cpPricingRules`，检查是否存在针对该终端客户的**特殊底价规则**。
2. 如果没有，则向上回溯，查询该 CP 的**全局底价规则**。
3. 根据找到的规则（固定、比例或阶梯），计算出 EG 应向 CP 收取的服务费金额。

### 3.3 步骤 3：解析 Layer 2 对客价
1. 引擎查询 `customerPricing`，检查 CP 是否为该终端客户配置了**特殊对客价**。
2. 如果没有，则使用 CP 在系统设置中配置的**默认加价策略**（例如：在底价基础上统一加收 $50）。
3. 计算出 CP 应向终端客户收取的最终服务费金额。

### 3.4 步骤 4：计算 CP Margin
引擎在内存中计算 `Margin = Layer 2 Fee - Layer 1 Fee`。
* **安全校验**：系统会进行强制校验，确保 `Layer 2 Fee >= Layer 1 Fee`。如果 CP 错误地将对客价设置得比底价还低，引擎会抛出异常并阻断发票生成，防止 CP 产生“倒贴”的亏损账单。

## 4. 特殊场景：直客模式与免收服务费

### 4.1 直客模式 (EG-DIRECT)
如果客户不属于任何外部 CP，而是 EG 直接签约的客户，引擎会跳过 Layer 1 的底价计算，直接根据 `customerPricing` 中的协议价计算出唯一的服务费，生成单层发票。

### 4.2 长期休假 (On Leave) 降费处理
对于处于长期休假（如产假）且当月无薪水发放的员工，系统通常不应收取全额的 EOR 服务费。
定价引擎支持配置 `leaveDiscountPercentage`。例如配置为 50%，则该员工在休假期间的服务费将减半收取。

通过高度参数化与层级化的设计，CP 定价引擎赋予了 Extend Global 平台极强的商业拓展能力，使其能够轻松适应全球各地、各种规模渠道伙伴的复杂商业诉求。
