# 双层发票引擎 (Dual-Layer Invoice Engine)

双层发票引擎是 Extend Global (EG) 平台 B2B2B 商业模型的核心基础设施。本文档详细阐述该引擎的设计理念、生成逻辑与数据结构，旨在帮助开发人员和产品经理深入理解平台如何处理多层级的计费需求。

## 1. 业务背景与设计初衷

在传统的 B2B 模式下，系统只需向终端客户开具一份账单。然而，在 EG 的 B2B2B 模式中，渠道合作伙伴 (CP) 并不希望终端客户知道 EG 的存在，也不希望客户知道底层的真实服务成本。

因此，每次薪酬结算周期结束时，系统必须基于同一份底层薪酬数据，**同时且自动地**生成两份相互独立的发票：

1. **Layer 1 Invoice (底层成本发票)**：由 EG 开具给 CP。包含实际的员工薪酬成本（Pass-through）以及 EG 向 CP 收取的底价服务费。这份发票仅在 Admin Portal 和 CP 的管理界面可见。
2. **Layer 2 Invoice (终端客户发票)**：由 CP 开具给终端客户 (Client)。包含相同的员工薪酬成本，但服务费部分被替换为 CP 设定的加价（Markup）后的金额。发票的抬头、Logo 和银行账户信息全部替换为 CP 的信息。这份发票在终端客户的门户中可见。

## 2. 核心计算逻辑与引擎工作流

双层发票的生成并非简单的金额相加，它涉及复杂的汇率转换、阶梯定价与最低收费逻辑。以下是引擎的标准工作流：

### 2.1 数据准备与分组
每月发薪日之前，系统会收集所有状态为 Active 的员工的薪资数据，包括基本工资、津贴、扣款、已审批的报销单以及雇主应承担的法定社保公积金。引擎首先按“终端客户 (Customer)”对这些数据进行分组。

### 2.2 成本基础计算 (Pass-through Costs)
对于每个客户分组，引擎计算所有员工在当地货币（如 EUR, GBP）下的总用工成本。这部分成本属于“代收代付（Pass-through）”性质，在 Layer 1 和 Layer 2 中**金额绝对保持一致**（转换为结算货币后）。

### 2.3 Layer 1 服务费计算 (EG ↔ CP)
引擎查询 `cpPricingRules` 表，获取 EG 为该 CP 设定的底价策略。
计费规则可能包括：
* **固定人头费**：如每位员工每月 $150。
* **按比例收费**：如总用工成本的 5%。
* **阶梯计费**：如 1-10 人每人 $150，11-50 人每人 $130。
* **最低消费 (Minimum Fee)**：如果按比例计算的金额低于最低消费，则按最低消费收取。

计算得出的服务费加上 Pass-through 成本，构成 Layer 1 发票的总金额。

### 2.4 Layer 2 服务费计算 (CP ↔ Client)
引擎查询 `customerPricing` 表（针对特定客户的协议价）或使用 CP 的默认加价策略。
CP 的加价策略通常是在 Layer 1 的基础上进行配置，例如：
* 在 EG 底价基础上固定增加 $50。
* 设定对客户的最终售价为每人 $250（系统自动计算差额）。

计算得出的加价服务费加上相同的 Pass-through 成本，构成 Layer 2 发票的总金额。

### 2.5 账单生成与持久化
引擎在 `invoices` 表中插入两条记录，分别标记 `invoiceLayer` 为 `eg_to_cp` 和 `cp_to_client`。同时，在 `invoiceItems` 表中插入详细的费用明细项，确保两份发票的数据在数据库层面建立关联，以便后续的资金流转与对账。

## 3. 关键数据结构与状态机

在数据库层面，发票实体包含以下关键字段以支撑双层架构：

### 3.1 核心字段说明
* `invoiceLayer`: 枚举值，区分 `eg_to_cp` (Layer 1) 和 `cp_to_client` (Layer 2)。对于 EG 直客，此字段为 `eg_to_client`。
* `channelPartnerId`: 关联的 CP 实体 ID。
* `customerId`: 关联的终端客户 ID。即使是 Layer 1 发票，也会记录该字段，以便追溯这笔成本是由哪个客户产生的。
* `settlementCurrency`: 发票的结算货币（通常为 USD 或 CP 指定的货币）。
* `totalAmount`: 发票总金额。
* `serviceFee`: 服务费金额（Layer 1 和 Layer 2 的差额即为 CP 的毛利）。

### 3.2 发票状态机
发票生成后，遵循严格的状态流转机制：
1. **DRAFT (草稿)**：引擎自动生成后的初始状态。此时允许 Admin 或 CP 进行手动调整。
2. **PUBLISHED (已发布)**：确认无误后发布。此时系统会生成 PDF 文件，并向相应的接收方发送邮件通知。
3. **PAID (已支付)**：款项结清。对于 Layer 2，由 CP 手动标记；对于 Layer 1，当 CP 从钱包中扣款成功后自动流转至此状态。
4. **VOIDED (已作废)**：发生严重错误需重新生成时使用。

## 4. 异常处理与边界条件

在实际业务中，双层发票引擎必须处理多种复杂的边界条件：

### 4.1 跨币种结算差异
如果终端客户要求以非 USD 货币（如 EUR）结算，而 EG 与 CP 之间约定以 USD 结算，引擎会在生成 Layer 1 时应用 EG 的内部汇率，在生成 Layer 2 时应用 CP 配置的对客汇率。这会产生汇兑损益，系统会通过专门的汇差剥离引擎进行核算。

### 4.2 直客模式兼容 (EG-DIRECT)
虽然系统主打 B2B2B 模式，但也支持 EG 直接签约的终端客户。在此场景下，系统会将该客户归属到一个虚拟的内置 CP（通常 ID 为 1，命名为 "EG-DIRECT"）。引擎检测到此特殊 CP 时，会跳过 Layer 1 和 Layer 2 的双层逻辑，直接生成单层的 `eg_to_client` 发票，避免数据冗余。

### 4.3 线下调整与红字发票 (Credit Notes)
如果发票发布后发现薪资计算有误（如员工少报了请假），系统不允许直接修改已发布的账单。此时，必须通过生成红字发票（Credit Note）来进行冲抵。引擎在生成 Credit Note 时，同样会遵循双层逻辑，分别生成对应 Layer 1 和 Layer 2 的红字凭证。

通过上述精密的设计，双层发票引擎在保证底层财务数据绝对准确的前提下，赋予了渠道合作伙伴极大的商业灵活性，是 Extend Global 平台最具核心竞争力的技术资产之一。
